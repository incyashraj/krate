How it works

From an agent's output to a running app — in three steps.

No installer to build, no operating system to pick, no source tree to trust. The file carries the app and its permissions together.

01

Get one file

An agent packages its output as a single .krate — the app plus the exact permissions it needs. Share it like any document, or link to it.

02

Double-click it

A window shows who's asking and everything it wants to touch — one folder, say. Nothing runs until you click Open. No terminal required.

03

It runs, sandboxed

The same file runs natively on macOS, Windows, and Linux. It starts with zero authority and gets only what you granted — nothing else on your machine is reachable.

The permission wall

Every capability is granted, never assumed.

Software has always run with your full authority — every file, every credential, the whole network. That was a shaky default when a person wrote the code. When an agent writes it and nobody reads it, it stops being defensible.

Grant — the app gets exactly the capabilities you approve, for that session only.
Deny — a withheld capability stops the app before a single line of its code runs, and returns a structured answer an agent can act on.

Proof, not promises

Every push, a machine nobody has logged into opens the app, clicks it, types into it, withholds a permission to confirm it refuses to start, and photographs the screen — on macOS, Windows, and Linux. It's all public CI. the same file, on every OS, unchanged.

Who it's for

Built for the moment software started writing itself.

Agent platforms

Your agents generate apps your users can't safely run on their own machines, so they stay trapped in cloud sandboxes and browser previews. Krate is the layer that lets you ship them locally, behind an enforced permission wall.

Anyone sharing a tool

Send a colleague a small tool the way you'd send a doc. They open it, see what it wants, and use it — without installing anything or trusting code they can't read.
